# 因子分析系统 · 建设方案

> 目标:做一个和模拟交易平级的「因子分析」模块。用户提交一条 SQL(查因子值),系统每天全量重跑,
> 算出 IC / ICIR / 分组收益 / 多空曲线等绩效,前端用列表页 + 详情页展示(样式已定稿见 `factor_demo.html`)。
>
> **双接口设计**:一个核心引擎,两种用法 ——
> - `analyze_factor()` 即时接口:notebook 里传 SQL 立即返回结果对象(可 `.plot_*()` 画图),不落盘、一次性,用于探索调参。
> - `submit_factor()` 提交接口:把因子登记入库,由每日定时任务 `run_all()` 全量重跑落盘,前端读结果展示。
>
> 二者共用同一个 engine,口径永远一致(notebook 里看到的 = 页面上显示的)。这与 simulation 里
> `run_backtest`(即时)vs `submit`(纳入调度)是同一套心智模型。

## 零、设计定稿(已和你确认过的决策)

| 决策点 | 结论 |
|--------|------|
| 因子值来源 | 用户传 **SQL**,必须返回 `date` / `instrument` / `value` 三列 |
| 时间范围 | **不指定起止**,每天**全量重跑**覆盖旧结果(无状态、无增量) |
| 未来收益 | **系统自己算**,不让用户在 SQL 里带收益列(防未来函数) |
| 收益口径 | 系统固定用**次日开盘价**:`open(T+1)` 买入 → `open(T+1+N)` 卖出,持有 N 日。买入价是算出因子之后才发生的价,零未来函数,最贴实盘。不暴露参数 |
| 股票池 universe | **砍掉该参数**。股票池由用户在 SQL 里自行控制(想要成分股就在 SQL 里 join 成分股表 WHERE 掉)。系统只做「可交易过滤」(剔停牌/ST/涨跌停/次新),这是能否成交的硬约束,与股票池选择正交 |
| 数据源 | 复用回测同一个 xxydb,**只依赖 `daily_bar` + `stock_status` 两张表** |
| 中性化 | **系统不做**。用户按需在 SQL 里自行处理(回归取残差 / 直接查已中性化因子值),系统不碰 exposure 表 |
| 并行 | 复用模拟交易的 `ProcessPoolExecutor` 模式,每因子独立目录,写入无竞争 |
| 存储 | 每因子独立目录,和 simulation 的「每账户独立目录」同一路子 |
| 计算策略 | **SQL-first**:能 SQL 就 SQL(用户因子SQL当子查询,IC/分组/收益下推 DuckDB),换手等 SQL 啃不动的才回落 pandas。以快为主(详见阶段1.6) |
| 多空方向 | **跟随 direction**(IC 符号自动判定或用户指定):long→做多高分位`Q_hi-Q_lo`;short→做多低分位`Q_lo-Q_hi`。多头组换手/年度多头收益同步跟随。这样负向因子(换手率等)多空年化转正,且方向依据是 IC 符号(统计规律)而非事后挑高组——无数据窥探(详见阶段1.6补丁) |

**开工前的待确认点已全部拍板,无阻塞,可直接从阶段 0 开工。**

## 一、整体架构(对齐现有 simulation 模块)

```
xxybacktest/
  factor/                      ← 新增模块,和 simulation/ 平级
    __init__.py                ← 对外导出 analyze_factor / submit_factor 等
    engine.py                  ← 引擎主入口 + pandas 收尾(换手/打包),兼容纯DataFrame回退版
    engine_sql.py              ← SQL 模板 + 拼接:把用户因子SQL当子查询,下推 IC/分组/收益到 DuckDB
    result.py                  ← FactorResult 结果对象(.summary()/.plot_ic()/.plot_groups())
    api.py                     ← analyze_factor() 即时接口(engine 的薄门面,不落盘)
    submitter.py               ← submit_factor / list / get / delete(元数据 CRUD)
    runner.py                  ← run_single(单因子) / run_all(并行全量,落盘)
    store.py                   ← 每因子独立目录的读写(meta/metrics/groups)
  web/
    routes/
      factor.py                ← 新增蓝图:列表页/详情页/API
    templates/
      factor_list.html         ← 列表页(移植 factor_demo.html)
      factor_detail.html       ← 详情页(移植 factor_demo.html)

data/
  factor_analysis/
    factors/
      <factor_id>/
        meta.json              ← 因子定义(name/sql/category/direction…)
        metrics.parquet        ← 汇总绩效(列表页读这个:IC均值/ICIR/多空年化…)
        ic_series.parquet      ← 逐日IC时序(详情页累计IC曲线)
        yearly.parquet         ← 年度拆解(详情页年度表)
        groups.parquet         ← 分组收益 + 净值曲线(详情页图)
```

**设计原则**:计算(engine)、调度(runner)、存储(store)、元数据(submitter)、展示(web)彻底分层,
每层可单独测试。engine 是纯函数,喂 DataFrame 出 DataFrame,不碰文件、不碰数据库连接。

**两个门面共用一个引擎**:
```
              engine.analyze(factor_df, price_df, status_df, ...)   ← 纯计算,内存进内存出
                          │
          ┌───────────────┴───────────────┐
   analyze_factor()                   submit_factor() + run_single()
   (api.py, 即时)                     (submitter/runner, 落盘)
   自己开db查SQL→喂engine             登记元数据,每日重跑→写因子目录
   返回 FactorResult(可画图)          结果落盘,前端读目录展示
   notebook 探索用                    生产监控用
```
口径修正只改 engine 一处,两个门面同步生效,永不出现"notebook 与页面对不上"。

### 阶段 1 · 计算引擎 engine.py(最核心,纯函数)

**目的**:把「因子值面板 + 行情」变成「IC + 分组绩效」。这是整个系统的心脏,和 web、存储完全解耦。

**输入**:
- `factor_df`:SQL 查出来的 `date / instrument / value`
- `price_df`:从 daily_bar 读的行情(`open` + `adjust_factor`,算后复权开盘收益)
- `status_df`:从 stock_status 读的 `suspended` / `st_status` / `price_limit_status`
- 参数:`forward_periods=[1,5,10,20]`、`n_groups=10`、`ic_method='rank'`

**内部步骤**(严格对应之前讲的流程):
1. 算未来 N 日收益(**次日开盘口径 + 后复权**:`(open×adj)(T+1+N) / (open×adj)(T+1) - 1`)→ 收益面板
2. 清洗票池(剔停牌/ST/涨跌停,**不剔次新**)
3. 横截面预处理(去极值 MAD + zscore)
4. 逐日算 rank IC → IC 时序
5. 分 N 组 → 每组收益、多空收益、净值曲线、换手
6. 汇总年度指标、全区间指标

**输出**(纯 DataFrame,不落盘):
- `ic_series`、`yearly`、`groups`、`metrics` 四张表

**验收**:用阶段 0 那个最简因子,喂进 engine,肉眼核对 IC 符号合理、分组单调、多空曲线走向正确。
配一个 `tests/test_factor_engine.py`,用构造数据验证"因子=未来收益本身时 IC≈1"。

**✅ 阶段1 完成(2026-07-09)**:
- `xxybacktest/factor/engine.py` 已实现:`compute_forward_returns`(次日开盘+后复权)、
  `build_tradable_mask`、`preprocess_cross_section`(MAD去极值+zscore)、`compute_ic_series`、
  `compute_groups`、`summarize`、主入口 `analyze()`。
- **metrics 已对齐前端 demo 全部字段**:ic_mean / icir / ic_win_rate / ls_return /
  ls_sharpe / ls_maxdd / **turnover(换手)** / **coverage(覆盖度)** / direction / 各周期IC。
- **年化口径已改严谨**:分组回测按 base_period **不重叠采样**(每 base_period 天调一次仓),
  净值累乘后 `nav^(252/持有天数)-1`,消除重叠样本导致的净值/回撤虚高。
- `tests/test_factor_engine.py` **8 项全过**:收益口径、完美因子IC≈1且单调、随机因子IC≈0、
  过滤生效、后复权消除跳变、换手+覆盖度存在且合理、半覆盖≈0.5、不重叠采样期数校验。
- 真实数据核对(昨日涨幅,2024-06起):IC=-0.016 判 short ✅;换手0.85(高频反转,合理)✅;
  覆盖度1.0 ✅;改严谨后多空回撤 -77%→-17%、夏普 -0.45→+0.97(修正了重叠导致的失真)。

### 阶段 1.6 · SQL-first 引擎重构(性能升级,深度绑定 xxydb)

**动机**:因子分析以**快**为主。阶段1的 pandas 引擎逻辑正确但有两个 Python 逐日循环
(`compute_ic_series`、`compute_groups`),因子量大时累积慢。**原则:能用 SQL 就用 SQL,
把用户因子 SQL 当子查询,计算层直接拼接下推到 DuckDB;实在啃不动的(换手)才回落 pandas。**

**核心手法**:用户的因子 SQL 原样塞进 CTE,外层套上"对齐收益 + 过滤 + rank + 分组 + 聚合"的 SQL,
一条大 SQL 在 DuckDB 里跑完绝大部分计算,`.df()` 出来的已是接近成品的小结果。

#### 各指标的 SQL 可行性分级(决定下推边界)

| 计算 | 归属 | SQL 手法 |
|------|------|----------|
| 未来收益(次日开盘后复权) | ✅ SQL | `LEAD(open*adj, 1/1+N) OVER (PARTITION BY instrument ORDER BY date)` |
| 可交易过滤 | ✅ SQL | `JOIN stock_status WHERE suspended=0 AND st_status=0 AND price_limit_status=2` |
| zscore 标准化 | ✅ SQL | 截面窗口 `(value-avg)/stddev OVER (PARTITION BY date)` |
| 逐日 rank IC(各周期) | ✅ SQL | 截面 `RANK() OVER(PARTITION BY date)` 后 `corr(rank_v, rank_r) GROUP BY date` |
| 不重叠调仓采样 | ✅ SQL | 日期编号 `dense_rank`,`WHERE (dnum-1) % base_period = 0` |
| 分组(NTILE)逐组收益 | ✅ SQL | `NTILE(n) OVER(PARTITION BY date ORDER BY value)` + `GROUP BY date, grp` |
| 覆盖度 | ✅ SQL | 两个 `COUNT(DISTINCT)` 之比 |
| 年度/全区间汇总 | ✅ SQL | `GROUP BY year` + 聚合;净值累乘用 `exp(sum(ln(1+r)))` |
| 最大回撤 | 🟡 SQL可 | 净值序列 `min(nav/cummax-1)`,窗口 `MAX() OVER` |
| MAD 去极值 | 🟡 回落pandas | DuckDB median 嵌套两层较绕;放 pandas 或先用 quantile 近似 |
| **换手率** | 🔴 pandas | 相邻调仓期**成分股集合差异**,SQL 需 array_agg+list_intersect,天书难维护 → 留 pandas |

#### 目标架构(engine 内部分两段)

```
┌─────────────────────────────────────────────────────────────┐
│  SQL 段(在 DuckDB 里,一条大 SQL,和用户因子 SQL 拼接)        │
│  WITH user_factor AS ( <用户SQL> ),                          │
│       aligned   AS ( 收益LEAD + 过滤JOIN + zscore窗口 ),      │
│       ranked    AS ( 截面 RANK / NTILE ),                    │
│       ...                                                    │
│  一次性产出:ic_series、group_returns、coverage、成分明细      │
└──────────────────────────┬──────────────────────────────────┘
                           │ .df()  (已是小结果)
┌──────────────────────────▼──────────────────────────────────┐
│  pandas 段(轻量收尾,只处理 SQL 不擅长的)                     │
│  · 换手率:对成分明细算相邻期集合差异                          │
│  · 汇总打包 metrics / yearly 成最终 dict                      │
└─────────────────────────────────────────────────────────────┘
```

#### 实现拆分

- `engine_sql.py`(新):存放 SQL 模板 + 拼接函数 `build_analysis_sql(user_sql, params)`,
  返回可直接 `db.query()` 的大 SQL。模板用参数占位(periods / n_groups / base_period / 过滤开关)。
- `engine.py`(改):`analyze()` 保留同样的入参与返回契约(**六键 dict 不变**),内部改为:
  ①调 `build_analysis_sql` → ②`db.query().df()` 拿 SQL 结果 → ③pandas 收尾算换手 + 打包。
- 新增一个 `analyze_from_sql(db, user_sql, params)` 作为 SQL 版主入口;
  旧的纯 DataFrame 版 `analyze(factor_df, price_df, status_df, ...)` **保留**,
  作为"已有 DataFrame / 单元测试 / 无 db 场景"的回退路径(两版共用 pandas 收尾函数)。

#### 兼容与测试策略(不破坏已有成果)

- **返回契约不变**:六键 dict(ic_series/groups/group_summary/ls_series/yearly/metrics)保持一致,
  下游 result.py / store.py / web 完全无感。
- **双版本对拍**:新增 `test_engine_sql_matches_pandas.py` —— 同一份构造数据,
  分别走 SQL 版和 pandas 版,断言 IC/分组/换手数值一致(容差 1e-6)。这样 SQL 版有了信任基础。
- 阶段1 的 8 个 pandas 单测继续保留(测 pandas 回退路径),不删。

#### 验收

- SQL 版与 pandas 版在真实数据(昨日涨幅、收盘价因子)上数值一致。
- 单因子耗时明显下降(逐日循环消除),记录 before/after 基准。

### 阶段 1.5 · 结果对象 result.py + 即时接口 api.py(notebook 门面)

**目的**:把 engine 的裸输出包装成好用的结果对象,并提供 `analyze_factor()` 即时接口。
让用户在 notebook 里一行代码分析一个因子,风格对齐现有 `BarraLens`(analyse.py)。

**FactorResult 结果对象**(result.py):持有 engine 输出的四张表,并提供:
- `.summary()` — 打印 IC均值 / ICIR / 多空年化 / 胜率 / 换手等核心指标
- `.plot_ic()` — 画累计 IC 曲线
- `.plot_groups()` — 画分组年化收益柱状 + 分组净值曲线
- `.plot_ls()` — 画多空净值曲线
- `.ic_series` / `.yearly` / `.groups` / `.metrics` — 直接取原始 DataFrame
- `.to_dict()` — 转成前端/落盘用的 JSON 结构(供 store 和 web 复用)

**analyze_factor() 即时接口**(api.py):engine 的薄门面,自己开一个只读 db 连接。
```python
from xxybacktest.factor import analyze_factor
res = analyze_factor(
    sql="SELECT date, instrument, close/pre_close-1 AS value FROM daily_bar",
    data_path=r"D:\Desktop\最新回测框架\data",
    forward_periods=[1,5,10,20], n_groups=10,
)
res.summary()
res.plot_groups()
```
职责:执行 SQL → 校验三列 → 读行情+状态 → 调 engine → 返回 FactorResult。**不落盘、不登记**。

**验收**:notebook / 命令行调 `analyze_factor` 传昨日涨幅因子,能打印 summary、能画三张图。

### 阶段 4 · 调度 runner.py(单因子 + 并行全量)

**目的**:串起 submitter(取定义)→ 读数据 → engine(算)→ store(存)。并提供并行全量重跑。

**函数**:
- `run_single(factor_id, data_path)`:取因子定义 → 复用 `analyze_factor` 的计算路径拿到 `FactorResult`
  → 用 `result.to_dict()` 经 store 落盘。**与即时接口共享同一 engine,口径必然一致**。
- `run_all(data_path)`:对所有因子并行重跑,**直接复用** simulation 的 `ProcessPoolExecutor` 写法。
  这也是**前端"每日更新"的真正入口**(定时任务调它),而非用户手动调 `analyze_factor`。

**验收**:`run_single` 能把一个因子从 SQL 一路算到落盘;`run_all` 多因子并行不丢数据、不报错。

**✅ 阶段4 完成(2026-07-09)**:
- `xxybacktest/factor/runner.py` 实现。`run_single(factor_id)`:读 meta → `analyze_from_sql`(与即时
  接口同一 engine)→ `FactorResult` → store 落盘 + 回填 meta status=ok;用户指定 direction 覆盖自动判定;
  失败置 status=error+last_error 且**保留上次结果**(不删)。`run_all()`:ProcessPoolExecutor 并行,
  每子进程自开 db(内存 DuckDB 无锁)+ 写独立目录,天然无竞争。前端"每日更新"入口。

  run_all 并行(2 因子 11s,Windows spawn 需 __main__ 保护)。全通过。
