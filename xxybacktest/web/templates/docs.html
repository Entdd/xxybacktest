系统概述

xxybacktest 是一套量化策略从研发到每日自动运行的完整闭环。用户写一次策略代码，注册为模拟账户后，系统每天自动重跑回测、更新净值曲线、生成当日交易信号。打开 Web 界面就能看到"今天该买什么卖什么"，直接跟单操作。

核心功能

  • 策略回测：基于历史数据验证策略表现
  • 模拟账户：将策略注册为每日自动运行的模拟交易账户
  • 实盘交易：对接 QMT，策略信号自动转化为真实委托下单
  • 因子分析：一条 SQL 提交因子，系统每日算 IC / 分组收益 / 多空曲线，看板监控
  • 定时任务：灵活调度数据更新、策略重跑等任意脚本
  • Web 可视化：策略净值、持仓、信号、因子绩效一目了然

架构简图

xxy-sim 启动
└── Flask (port 5000)
    ├── /              ← 账户列表（模拟+实盘）
    ├── /account/<id>  ← 账户详情（净值曲线、持仓、信号）
    ├── /factors       ← 因子绩效看板（列表）
    ├── /factors/<id>  ← 因子详情（年度IC、分组收益、净值曲线）
    ├── /tasks         ← 定时任务管理（含实盘调度）
    └── /docs          ← API 文档（本页）

APScheduler（后台）
    ├── builtin_run_simulation  ← 内置：每日模拟交易重跑
    ├── builtin_run_factors     ← 内置：每日因子分析全量重跑
    ├── live_<account_id>      ← 实盘：定时触发策略下单（QMT）
    ├── task_数据更新            ← 用户注册：数据更新脚本
    └── task_xxx                 ← 用户注册：任意脚本

实盘链路
    APScheduler → live/runner.py → QMTTrader → xtquant → 真实委托

回测用法

编写策略代码

策略由两个函数组成：initialize 和 handle_data。

from xxybacktest.simulation import submit

def initialize(context):
    """初始化函数，策略注册时执行一次"""
    # 加载历史数据、设置参数等
    pass

def handle_data(context):
    """每日调仓函数，每天收盘后执行一次"""
    # 选股、调仓等逻辑
    # 示例：全仓买入
    context.order_target_percent("000001.SZ", 1.0)

运行单次回测

策略编程接口

Context 对象核心属性

属性说明

下单函数

所有下单函数均已绑定到 context，在策略中直接通过 context.xxx() 调用。

函数说明
context.order(security, amount) 按数量下单。amount > 0 买入，amount < 0 卖出，0 不下单
context.order_buy(security, amount) 买入指定数量（股）
context.order_sell(security, amount) 卖出指定数量（股）
context.order_value(security, value) 按金额下单。value > 0 买入对应金额，value < 0 卖出对应金额
context.order_target_value(security, value) 调仓至目标市值。value=0 表示清仓
context.order_target_percent(security, percent) 按总资产百分比调仓。percent 为 0.0~1.0 小数，如 0.1 表示 10%
context.inout_cash(amount) 出入金。正数入金，负数出金

数据接口

# 获取最新价格（回测/实盘统一接口，返回 float 或 None）
price = context.get_price(security)
#   回测：按当前时间返回模拟撮合价（盘前=上一日close，盘中=当日open，盘后=当日close）
#   实盘：返回 QMT 实时 tick 价
#   停牌 / 无行情时返回 None，使用前需判空
price = context.get_price("000001.SZ")
if price:
    context.order_target_percent("000001.SZ", 0.10)

# 获取历史行情（防未来数据：只回溯到上一交易日，不含当日）
his = context.history(instruments, fields=None, bar_count=1)
#   instruments: 股票代码列表，如 ["000001.SZ", "000002.SZ"]
#   fields:      字段列表，默认 ["close"]；可选 open/high/low/close/pre_close/volume/amount/vwap
#   bar_count:   回溯的K线数量
#   返回:        dict {code: np.recarray}，按属性访问列

his = context.history(["000001.SZ"], ["close", "volume"], 10)
arr = his.get("000001.SZ")          # 先判空，可能返回空 dict
if arr is not None:
    last_close = arr.close[-1]      # 最近一根（上一交易日）收盘价
    closes = arr.close              # float64 数组
    dates = arr.date                # 日期字符串数组

实盘交易

实盘账户通过 APScheduler 定时触发 run_live()，自动连接 QMT 执行策略并产生真实委托。策略代码与回测完全一致，无需修改。

实盘执行流程

  1. APScheduler 按 trigger_cron 触发 run_live(account_id)
  2. 连接 QMT，读取当前持仓和总资产
  3. 从数据库加载最新策略源码，构建实盘 context
  4. 执行 initialize（首次）和 handle_data
  5. 策略中的下单函数（如 order_target_percent）直接发往 QMT
  6. 保存调仓结果（净值、持仓、订单）到 data/live/accounts/<id>/
  7. 持久化策略状态 context.g 到 live_schedule.json

策略编写注意事项

关键规则：策略函数内部不要引用外部全局变量。 run_live() 用 exec() 在独立模块中加载策略源码，外部变量不可见。 所有配置应在函数内部硬编码，或通过 context.g 持久化存储。 
# 错误示范 ❌
TARGET_PCT = 0.10  # 外部全局变量

def handle_data(context):
    context.order_target_percent("600519.SH", TARGET_PCT)  # NameError!

# 正确示范 ✅
def handle_data(context):
    target_pct = 0.10  # 内部硬编码
    context.order_target_percent("600519.SH", target_pct)

# 或者用 context.g 持久化（推荐）✅
def initialize(context):
    context.g.setdefault("target_pct", 0.10)
    context.g.setdefault("run_count", 0)

def handle_data(context):
    context.g["run_count"] += 1
    context.order_target_percent("600519.SH", context.g["target_pct"])

策略状态持久化

context.g 支持基础类型（int、float、str、list、dict），每次 run_live 执行结束后自动保存到 live_schedule.json，下次执行前自动恢复。

# initialize 中初始化状态
context.g.setdefault("signal", "hold")
context.g.setdefault("positions", {})

# handle_data 中读取和更新
if context.g["signal"] == "buy":
    context.order_target_percent("600519.SH", 0.10)
    context.g["signal"] = "hold"

实盘交易函数

实盘 context 上的交易函数与回测签名完全一致：

函数说明
context.order(security, amount)按数量下单，自动转发到 QMT
context.order_target_percent(security, percent)按总资产百分比调仓（推荐）
context.order_target_value(security, value)调仓至目标市值
context.get_portfolio()实时从 QMT 拉取最新资金概况，同时同步到 context.portfolio。返回 {'cash', 'frozen_cash', 'market_value', 'total_asset'}
context.get_account_positions()实时从 QMT 拉取全部持仓，同时同步到 context.portfolio.positions。返回 {code: {'volume', 'can_sell_volume', 'cost_price', 'last_price', 'market_value'}}
context.get_price(security)获取最新价（实盘=QMT 实时 tick，回测=模拟撮合价）。返回 float，停牌/无行情返回 None。与回测接口一致
提示：get_portfolio() 和 get_account_positions() 在每次 handle_data 结束后会自动调用一次刷新 context。策略内部如需在盘中获取最新资金和持仓（如下单前校验），可手动调用。

# 修改策略代码（传入函数对象，自动提取源码）

因子分析

因子分析模块让你用一条 SQL 提交一个因子，系统自动计算 IC / ICIR / 分组收益 / 多空曲线等绩效指标，并在 /factors 看板展示、每日自动更新。适合回放大量因子、持续监控因子绩效衰减。

两种用法，共用同一套计算引擎，口径永远一致：
① analyze_factor(...) —— notebook 里即时分析，返回结果对象可画图，不落盘、一次性，用于探索调参。
② submit_factor(...) —— 提交入库，由每日定时任务全量重跑落盘，前端看板监控。

核心约定

  • 因子值来源：你的 SQL，必须返回 date / instrument / value 三列。
  • 收益口径：系统固定用次日开盘价 + 后复权算未来 N 日收益（open(T+1) 买入 → open(T+1+N) 卖出），零未来函数，最贴实盘。
  • 可交易过滤：系统自动剔除停牌 / ST / 涨跌停（次日无法成交的股票）。
  • 股票池 & 中性化：由你在 SQL 里自行控制（要成分股就 join 成分股表，要中性化就自己回归取残差 / 直接查已中性化的因子值）。
  • 方向判定：默认按 IC 符号自动判定（IC 为负=负向因子）。多空组合跟随方向：正向做多高分位、负向做多低分位，故负向因子多空年化也为正。

① 即时分析（notebook 探索）

from xxybacktest.factor import analyze_factor

# 一条 SQL 定义因子，立即拿到绩效
res = analyze_factor(
    sql="SELECT date, instrument, close/pre_close-1 AS value "
        "FROM daily_bar WHERE date >= '2023-01-01'",
    data_path=r"D:\数据目录\data",
    name="昨日涨幅",
    periods=[1, 5, 10, 20],   # 未来 N 日收益
    n_groups=10,              # 分组数
    base_period=5,            # 分组/多空/汇总用哪个周期
)

res.summary()        # 打印 IC均值/ICIR/多空年化/胜率/换手/年度拆解
res.plot()           # 四张图：累计IC / 多空净值 / 分组年化 / 分组净值
res.plot_ic()        # 单独画累计 IC 曲线
res.plot_groups()    # 单独画分组年化 + 分组净值
res.metrics          # 原始指标 dict
res.ic_series        # 逐日 IC 时序 DataFrame
res.to_dict()        # 转 JSON（落盘/前端用的结构）

因子逻辑可以直接写在 SQL 里，包括窗口函数。例如 5 日换手率均值：

sql = """
SELECT date, instrument,
       avg(turn) OVER (PARTITION BY instrument ORDER BY date
                       ROWS BETWEEN 4 PRECEDING AND CURRENT ROW) AS value
FROM daily_bar WHERE date >= '2023-01-01'
"""
res = analyze_factor(sql, data_path=r"D:\数据目录\data", name="5日换手率", base_period=5)

    category="价值",                  # 分类，列表页筛选用（必填）
    data_path="./data",
    periods=[1, 5, 10, 20],
    n_groups=10,
    base_period=5,
    ic_method="rank",                # rank(Spearman) / normal(Pearson)
    direction=None,                  # 不传按IC符号自动判定；也可指定 'long'/'short'
    description="市盈率倒数",         # 可选说明
    run_now=False,                   # True=提交后立即算一次；False=等定时任务
)
print(f"因子已登记: {factor_id}")

每日自动更新

启动 xxy-sim 后，因子分析会作为内置任务 builtin_run_factors 每日自动全量重跑。触发时间默认比模拟交易（--time）晚 30 分钟（因子分析依赖当天行情已入库），也可用 --factor-time 单独指定：

# 模拟交易 22:00 跑，因子分析 22:45 跑
xxy-sim --data ./data --time 22:00 --factor-time 22:45

# 不指定 --factor-time：因子分析默认 22:30（22:00 + 30 分钟）
xxy-sim --data ./data --time 22:00
提交过的因子会在每天定时触发时自动重算、刷新看板，无需手动干预。打开 http://localhost:5000/factors 即可查看所有因子的最新绩效。

绩效指标说明

指标含义
IC 均值因子值与未来收益的截面相关性均值，衡量因子整体有效性
ICIRIC 均值 / IC 标准差 × 年化系数，衡量因子的稳定性（性价比）
IC 胜率IC 为正（或负向因子为负）的截面日占比
多空年化多头组减空头组的年化收益（方向跟随因子 direction）
多空夏普多空组合的年化夏普比率
最大回撤多空组合净值的最大回撤
换手多头组相邻调仓期的成分股换手率
覆盖度因子有值的可交易股票占全体可交易股票的比例

定时任务

系统使用 APScheduler 管理定时任务，支持热注册、热删除，无需重启服务。

内置任务：每日模拟交易

启动 xxy-sim 时自动注册，通过 --time 参数控制触发时间：

xxy-sim --data ./data --time 22:00   # 每晚 22:00 自动重跑所有运行中账户
xxy-sim --data ./data --time 15:30   # 收盘后 15:30 执行

注册自定义定时任务

from xxybacktest.simulation import schedule_task

# 方式一：每天固定时间
tid = schedule_task(
    name="数据更新",
    script="./data_renew.py",   # 脚本的绝对或相对路径
    time="17:30",                # 每天 17:30 执行
    data_path="./data",
)

# 方式二：cron 表达式
tid = schedule_task(
    name="季报数据",
    script="./update_quarterly.py",
    cron="0 10 1 1,4,7,10 *",    # 每季度第一天 10:00
    data_path="./data",
)
参数说明
name任务名称（展示用）
scriptPython 脚本路径
time每天固定时间，格式 HH:MM，与 cron 二选一
cron标准 5 段 cron 表达式（分 时 日 月 周）
data_path数据目录路径
注意：time 和 cron 只能指定一个。相同 name + script + cron 的任务不会重复注册。

常见问题

Q：Flask 的 debug 模式可以开吗？

A：不可以。APScheduler 的 BackgroundScheduler 在 debug=True 时会被 reloader 启动两次，导致调度器重复运行。main.py 已固定 debug=False。

Q：手动触发任务时提示"任务正在执行中"怎么办？

A：同一任务设置了 max_instances=1，如果上次还没跑完，手动触发会被拒绝。请等待上次执行完成，或查看日志确认进度。

Q：Windows 下并行执行有问题吗？

A：Windows 使用 spawn 模式启动子进程，功能正常但启动开销比 Linux fork 大。如果账户数很少（<4），并行收益不明显。

Q：日志在哪里看？

A：任务日志存储在 data/simulation_results/task_logs/<task_id>.log，可以通过 Web 界面的"日志"按钮实时查看。

Q：实盘策略报错 "name 'xxx' is not defined" 怎么办？

A：策略函数内部引用了外部全局变量。实盘用 exec() 在独立模块中加载源码，外部变量不可见。请将所有配置硬编码在函数内部，或通过 context.g 持久化。

Q：修改实盘策略后需要重启服务吗？

A：不需要。调用 update_account() 修改策略代码或 cron 后，下次定时触发自动生效。APScheduler 的调度时间也会热更新。

Q：删除实盘账户后任务管理还有记录？

A：正常删除会同步清理 APScheduler job、live_schedule.json、结果目录和日志。如果仍有残留，运行 python scripts/cleanup_live_residual.py "D:\Desktop\最新回测框架\data" 清理。

Q：因子分析可以指定每日运行时间吗？

A：可以。用 --factor-time HH:MM 单独指定；不指定则默认比模拟交易（--time）晚 30 分钟。因子分析依赖当天行情已入库，所以应晚于模拟交易触发。

Q：因子的 IC 是正的，但多空年化是负的，是 bug 吗？

A：不是。这说明因子非单调——IC 衡量整体线性相关，多空只看最高/最低两个极端分位。当因子呈倒 U 形（如中间分位收益最高），两端对冲反而不赚钱，多空为负如实反映了"IC 显著但极端分位无 alpha"。这正是多空指标的价值。

Q：为什么因子值计算要自己写 SQL，不能传函数？

A：SQL 直接下推到 DuckDB 计算，比 Python 逐日循环快得多（实测 2 倍以上），且你最懂自己的数据表。股票池筛选、中性化等也一并在 SQL 里灵活处理，系统只负责收益口径、过滤和 IC/分组计算。

Q：一台笔记本能跑多少因子？

A：绩效指标每因子仅几十 KB，几百上千个因子的存储毫无压力。计算上 SQL-first 引擎单因子几秒，run_all 并行重跑几百因子也是分钟级。瓶颈在因子数量而非单因子速度，靠并行解决。

Q：数据存储格式变更后旧数据怎么办？

A：最简单的方式是让所有账户重跑一次，新文件会自动生成。模拟账户路径为 data/simulation_results/accounts/<account_id>/，实盘账户路径为 data/live/accounts/<account_id>/。
