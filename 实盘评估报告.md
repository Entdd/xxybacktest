# xxybacktest 实盘交易评估报告

> 评估日期：2026-05-22
> 评估范围：实盘交易模块（live/）、调度层（simulation/）、Web 层（web/routes/）
> 结论：**可用于小资金实盘测试，不建议大资金生产使用**

---

## 评估维度总览

| 维度 | 评分 | 说明 |
|------|------|------|
| 交易执行与订单管理 | 5/10 | 基础下单功能具备，但无成交确认、无订单跟踪 |
| 风险控制与合规 | 3/10 | 几乎无交易前风控，无持仓/资金校验 |
| 系统可靠性与容错 | 4/10 | 防并发机制脆弱，无断线重连，无超时控制 |
| 数据管理与一致性 | 6/10 | Parquet 存储简洁，但净值计算未考虑出入金 |
| 安全性与权限控制 | 2/10 | 无认证、无加密、策略代码可任意注入执行 |
| 监控与可观测性 | 4/10 | 仅有基础日志和 Web 展示，无告警机制 |
| 性能与扩展性 | 6/10 | 并行回测支持，但 Flask 内置服务器性能有限 |
| 策略开发与测试 | 7/10 | API 与回测兼容，但策略执行无沙箱隔离 |
| **综合评分** | **4.6/10** | **不建议直接用于生产实盘** |

## 维度一：交易执行与订单管理

### 现状

- 通过 QMT（xtquant）接入券商交易通道，支持市价/限价下单
- 提供与回测兼容的 API：`order_buy`、`order_sell`、`order_target_percent` 等
- 使用五档即成剩撤市价单，保证成交速度

### 不足之处

1. **无成交确认机制**
   - 下单后仅固定 `sleep(0.5)`，不轮询订单状态
   - `Order` 对象的 `status` 只有 `submitted`，无法区分"已报/部成/全成/已撤"
   - 策略以为成交的订单可能实际未成交，导致后续调仓基于错误假设

2. **无订单生命周期跟踪**
   - 没有订单表记录委托→成交的完整状态流转
   - 部分成交场景完全未处理
   - 无法支持撤单、改单操作

3. **价格获取与下单分离**
   - `order_buy()` 先 `get_price()` 获取价格用于记录，但实际用市价单下单
   - 若价格获取失败（返回 None），仍继续下单，记录的价格为 None

4. **市价单滑点风险**
   - 统一使用五档即成剩撤，流动性差的股票可能产生大幅滑点
   - 无滑点预估和限价保护机制

5. **无交易时间校验**
   - 未校验是否在连续竞价时段内
   - 集合竞价、盘后固定价格交易时段未区分

## 维度二：风险控制与合规

### 现状

- 模拟回测层面有仓位限制，但实盘交易函数中无对应风控
- 依赖 QMT 拒绝超限订单（被动风控）

### 不足之处

1. **无交易前风控检查**
   - 买入前不检查可用资金是否充足
   - 卖出前不检查可卖持仓（T+1 限制）
   - 不检查个股持仓集中度上限
   - 不检查总仓位上限

2. **无涨跌停/停牌检查**
   - 不判断标的是否涨停（无法买入）或跌停（无法卖出）
   - 不判断标的是否停牌
   - 策略可能向这些标的下单，导致订单被废单

3. **无 ST/*ST/退市整理期检查**
   - 策略可能买入风险警示股票，合规层面存在问题

4. **无异常交易监控**
   - 无自成交、频繁报撤、拉抬打压等异常交易行为检测
   - 监管合规风险高

5. **无单日亏损/回撤熔断**
   - 当单日亏损超过阈值时，无自动暂停交易机制
   - 极端行情下可能连续亏损

6. **无策略层面的仓位硬限制**
   - 策略通过 `order_target_percent()` 调仓，但无全局最大仓位约束
   - 策略 bug 可能导致满仓单票或过度杠杆

## 维度三：系统可靠性与容错

### 现状

- QMT 连接有 5 次重试机制（间隔 3 秒）
- 实盘账户有防并发文件锁（`live_schedule.json`）
- `run_live()` 有 `try/finally` 保证锁释放

### 不足之处

1. **防并发机制脆弱**
   - 使用 JSON 文件标记 `running` 状态，进程崩溃后可能永远为 `True`
   - 无超时自动释放机制，账户可能永久被跳过
   - JSON 文件读写无文件锁，多进程同时操作可能损坏

2. **无断线重连机制**
   - QMT 连接在策略执行过程中断开，整个 `run_live` 异常退出
   - 无运行时心跳检测，无法及时发现连接异常
   - 无自动重连和状态恢复逻辑

3. **策略执行无超时控制**
   - `initialize()` 和 `handle_data()` 无执行时间上限
   - 策略死循环或耗时操作会导致账户永远处于 running 状态
   - 无"慢策略"检测和强制终止

4. **异常时状态不一致风险**
   - `finally` 中设置 `running: False`，但如果 `_update_schedule()` 也失败，状态永久异常
   - 无状态校验和修复工具

5. **无盘中异常中断恢复**
   - 若 `run_live` 执行中被 kill，已下订单无法自动撤回
   - 无"安全模式"（异常时自动清仓或暂停后续交易）

6. **回调函数为空**
   - `XtQuantTraderCallback` 未实现任何回调
   - 无法接收成交回报、资金变动、连接断开等异步推送

7. **无多交易通道支持**
   - 仅支持 QMT，若 QMT 崩溃或券商通道故障，系统完全停摆
   - 无备用通道切换机制

## 维度四：数据管理与一致性

### 现状

- 模拟结果改为每账户独立 Parquet 文件，天然无竞争
- 实盘结果也使用独立目录存储（`live/accounts/{id}/`）
- 净值、持仓、订单分别存储

### 不足之处

1. **净值计算未考虑出入金**
   - `nav = total_value / starting_cash`，使用 submit 时的初始资金
   - 若期间有出入金、分红送股，净值曲线严重失真
   - 无资金变动事件记录和净值复权机制

2. **结果存储非原子性**
   - `_save_results()` 分三次写入不同 Parquet 文件
   - 若进程在写入中间崩溃，可能产生不一致状态（有净值无持仓等）

3. **无数据校验机制**
   - 写入 Parquet 前不校验数据完整性（空值、类型、范围）
   - 不校验读取到的数据是否与 QMT 一致

4. **历史数据预加载策略粗糙**
   - 预加载从 `start_date` 到今天的全部日线数据
   - 若 start_date 很早（如 10 年前），内存消耗巨大
   - 仅支持日线，分钟线等其他频率直接报错

5. **交易日判断依赖本地数据库**
   - `is_trading_day()` 查询本地 `trading_days` 表
   - 若数据库未更新或缺失，可能错误判断非交易日为交易日

6. **持仓数据无版本控制**
   - 每次调仓覆盖持仓快照，无历史版本
   - 无法追溯某时刻的精确持仓状态

## 维度五：安全性与权限控制

### 现状

- Web 服务使用 Flask，监听 0.0.0.0:5000
- 无登录、无 API Key、无 IP 白名单

### 不足之处

1. **API 完全无认证**
   - 任何人可访问 `http://ip:5000` 操作实盘账户
   - 可暂停、恢复、删除、修改策略代码、触发调仓
   - 等同于把交易终端暴露在公网

2. **策略代码任意注入执行**
   - `update_account_route()` 通过 API 直接修改 `initialize_code` 和 `handle_data_code`
   - `_load_func()` 使用 `exec()` 执行用户提供的策略源码，无沙箱隔离
   - 恶意代码可：读取/删除文件、访问网络、执行系统命令、窃取资金

3. **敏感信息明文存储**
   - `live_account_id` 和 `qmt_path` 以明文存储在数据库中
   - 无加密、无脱敏、无访问控制

4. **SQL 注入风险**
   - `account.py` 中 `_fill_missing_names()` 直接拼接 SQL 字符串
   - 若持仓数据被篡改，可通过注入修改或删除数据库

5. **Flask 生产环境不安全**
   - 使用 Flask 内置服务器（非 WSGI），高并发下性能和安全不足
   - 无 HTTPS，账户信息、策略代码在网络中明文传输
   - `debug=False` 虽关闭，但无请求频率限制，易受 DDoS

6. **无操作审计日志**
   - 暂停、删除、更新等关键操作无记录
   - 出现问题无法追溯责任人

7. **无数据备份机制**
   - 账户配置、策略状态、交易结果均无自动备份
   - 磁盘损坏或误操作导致数据永久丢失

## 维度六：监控与可观测性

### 现状

- 有基础日志输出（print）
- Web 界面可查看净值、持仓、订单
- APScheduler 有任务日志归档

### 不足之处

1. **无告警机制**
   - 交易失败、连接断开、策略异常均无通知（邮件/短信/微信/钉钉）
   - 只能在手动查看 Web 或日志时发现问题
   - 夜间定时任务失败无人知晓

2. **日志级别混乱**
   - 大量使用 `print()` 而非标准 logging
   - 无日志分级（DEBUG/INFO/WARNING/ERROR/CRITICAL）
   - 并行执行时各账户日志交错，难以阅读

3. **无关键指标监控**
   - 无实时跟踪：当日收益率、最大回撤、胜率、盈亏比等
   - 无账户健康度评分
   - 无策略表现偏离回测的检测

4. **无交易对账**
   - 系统记录的订单与 QMT 实际成交无自动对账
   - 数据不一致无法及时发现

5. **无系统资源监控**
   - 不监控 CPU、内存、磁盘、数据库连接数
   - 资源耗尽可能导致交易失败

6. **调度任务状态不可靠**
   - `is_job_running()` 通过文件状态判断，进程崩溃可能留下"running"状态
   - 无任务执行耗时统计和 SLA 监控

## 维度七：性能与扩展性

### 现状

- 模拟回测使用 `ProcessPoolExecutor` 并行执行
- Web 使用 `threaded=True` 处理并发请求

### 不足之处

1. **Flask 内置服务器性能瓶颈**
   - 生产环境应使用 Gunicorn/uWSGI + Nginx
   - 内置服务器单进程处理请求，高并发下响应慢

2. **历史数据预加载内存消耗大**
   - 每个实盘账户预加载全量历史数据到内存
   - 多账户同时运行时内存占用线性增长

3. **QMT 连接无池化**
   - 每次调仓新建连接、获取数据后断开
   - 频繁连接/断开增加延迟和资源消耗

4. **无数据库连接池**
   - `get_db()` 使用全局缓存，但无连接池管理
   - 高并发下可能耗尽连接资源

5. **session_id 冲突风险**
   - `random.randint(100000, 999999)` 范围较小
   - 多账户同时连接 QMT 可能 session_id 冲突

6. **无水平扩展能力**
   - 所有组件运行在同一进程内
   - 无法将交易执行、Web 服务、调度器分离部署

## 维度八：策略开发与测试

### 现状

- 实盘 API 与回测 API 签名兼容，策略可无缝切换
- 支持 `initialize` + `run_daily` 模式
- 策略状态可持久化（`context.g`）

### 不足之处

1. **策略执行无沙箱隔离**
   - `exec()` 执行策略代码，与主进程共享全局命名空间
   - 一个策略的异常可能影响其他策略或整个系统
   - 无资源限制（CPU、内存），恶意策略可耗尽系统资源

2. **无实盘模拟（Paper Trading）**
   - 无模拟券商环境，策略只能直接上实盘或纯回测
   - 无法在生产环境用模拟资金验证策略逻辑

3. **策略状态序列化限制**
   - `_serialize_state()` 处理 numpy/pandas 类型，但自定义对象会被 `str()` 转换
   - 反序列化后自定义对象类型丢失，可能导致策略行为异常

4. **无策略版本管理**
   - 修改策略代码后无版本历史
   - 无法对比不同版本的表现
   - 无法回滚到历史版本

5. **无 A/B 测试支持**
   - 无法同时运行多个策略变体对比表现
   - 策略优化依赖人工切换和观察

6. **无策略参数优化框架**
   - 无内置参数扫描、网格搜索、遗传算法等优化工具
   - 参数调优依赖外部脚本

7. **无策略表现归因**
   - 无法分析收益来源（行业、风格、个股选择、择时）
   - 无 Brinson 归因、Barra 风险模型等工具

- [ ] 实现交易前风控（资金/持仓/涨跌停/停牌检查）
- [ ] 实现订单状态轮询或成交回调
